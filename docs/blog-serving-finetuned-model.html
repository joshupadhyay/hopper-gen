Fine-tuning Stable Diffusion

Edward Hopper, LoRA, and fine-tuning is only 40%

Josh Upadhyay · March 2026

What's fine-tuning?

Large image generation models like Stable Diffusion XL ship with broad, general-purpose knowledge — they can render nearly any scene you describe. But "general" means "generic." If you want a model that consistently captures a specific visual style, prompting gets you only so far. You can describe the style you want in the prompt and get decent results, but some visual signatures — the way light pools on a countertop, the specific geometry of a lonely urban scene — are hard to encode in text. Fine-tuning bakes that knowledge directly into the model weights.

Why fine-tune at all?

Prompting is powerful. You can write "in the style of Edward Hopper" and get something Hopper-adjacent. But there are limits: the model's concept of "Hopper" is whatever its pretraining dataset contained, which may be shallow or inconsistent. Fine-tuning on a curated set of actual Hopper paintings teaches the model specific patterns — how he handled lamplight, how his compositions frame solitude, how his palette shifts between indoor and outdoor scenes. The model doesn't just reference the style; it internalizes it.

LoRA: the core idea

Instead of updating all 6.6 billion parameters during training, LoRA (Low-Rank Adaptation) freezes the pretrained weights and injects small trainable matrices into specific layers. These matrices are low-rank decompositions — a weight update matrix W is approximated as the product of two much smaller matrices A and B, where A is (d × r) and B is (r × d), and r (the rank) is tiny compared to d.

The result: you train roughly 0.1% of the total parameters, use a fraction of the memory, and the resulting adapter file is small enough to swap in and out at inference time. At deployment, the adapter weights merge into the base model, and the pipeline behaves as if it were a single fine-tuned model.

Edward Hopper as the target

The target style is Edward Hopper — the American realist painter known for warm lamplight, lonely diners, cinematic urban geometry, and that unmistakable sense of stillness. The training set was a collection of Hopper paintings and reproductions, and the goal was to get SDXL to reliably produce images that feel like they belong in his portfolio. Not pastiche — internalized understanding of how Hopper saw light and space.

Performance & Optimization

Training a model is only half the story. The other half is serving it — getting images generated fast enough that the experience feels responsive. This section walks through the optimization journey: what we tried, what worked, what didn't, and where we landed.

Cold starts and warm starts

When you deploy a model on serverless GPU infrastructure like Modal, your code doesn't run on a persistent server. Instead, a container spins up on demand, loads the model into GPU memory, and then handles requests. The first request after that spin-up is the cold start — it includes container initialization, model loading, and the first inference pass. Subsequent requests to the same running container are warm — the model is already in memory and ready to go. Optimizing both matters, but they respond to different techniques.

Cold start progression

The first optimization target was cold start time — how long a user waits when no container is running.

The biggest single win was Modal snapshots, which saved 16 seconds off the cold start. A GPU memory snapshot serializes the entire model state — weights, CUDA context, allocated buffers — into a checkpoint. When a new container starts, it restores from this snapshot instead of re-downloading and re-initializing the model from scratch. Think of it as hibernation for GPU state.

Baking the LoRA weights directly into the snapshot (rather than loading them dynamically at startup) saved an additional 2 seconds. The snapshot now contains the fully-merged model, ready to generate on the first request.

Warm latency progression

Once the container is running, the question becomes: how fast can we generate an image? This is the latency users feel on every request after the first.

The fastest warm time we measured was 17.7 seconds using tf32 precision with compile_repeated_blocks() — a technique that applies torch.compile selectively to the repeated UNet blocks rather than compiling the entire pipeline. But we didn't ship that configuration.

The fastest configuration wasn't chosen as the production default. After a snapshot restore, the first request with compiled blocks was unpredictable — sometimes fast, sometimes slower than baseline as the compiled cache warmed up. Production favors consistency over peak speed.

The cold/warm tradeoff

Every optimization decision involves a tradeoff between cold and warm performance. The scatter plot below maps each configuration we tested onto these two axes — the ideal position is the bottom-left corner: fast cold start and fast warm latency.

The outlier is immediately obvious: bf16 + channels_last + compile at 99.45 seconds cold. What happened? torch.compile traces and optimizes the computation graph at runtime, producing highly efficient CUDA kernels. But when LoRA adapter weights are present, they alter the graph structure. Each time the container restores, the compiled cache is invalidated and the entire model must be recompiled from scratch — a process that takes over a minute on SDXL. This made the combination of torch.compile and LoRA fundamentally incompatible for production serving.

Production latency profile

The final deployed configuration hits three distinct latency tiers depending on the container state.

After a fresh deploy, the first request is the fastest cold start because Modal creates the snapshot eagerly. After a container scales to zero and restores, the snapshot restore path adds a small overhead. Warm requests settle into a tight, predictable band.

Warm variance

Consistency matters as much as raw speed. Here are three consecutive warm requests showing the spread:

A 0.25-second spread across warm requests. The pipeline is deterministic enough that variance comes from GPU scheduling jitter, not from anything in our code. This is the kind of stability that makes the production config worth the marginal speed sacrifice.

What we settled on and why

The final production configuration balances speed, consistency, and compatibility:

  • fp16 base weights with the fp16-safe VAE decoder (avoids NaN outputs in half-precision)
  • tf32 enabled for matrix multiplications — free speedup on Ampere+ GPUs with no precision loss for inference
  • channels_last memory format — matches the GPU's preferred memory layout for convolutions
  • LoRA weights loaded and merged into the snapshot, not loaded dynamically at startup
  • A warm-up inference pass runs before the snapshot is captured, so CUDA kernels and memory pools are pre-allocated
  • No torch.compile — incompatible with LoRA weight merging, causes full recompilation on restore
  • 50 inference steps (quality-optimized; speed-optimized configs use 25-30)

SDXL was overkill

Let's be honest: SDXL is a 6.6 billion parameter model designed to generate photorealistic, publication-quality images at 1024×1024. For this project — generating Hopper-style paintings for a personal site — that's almost certainly more model than needed. A smaller architecture like Stable Diffusion 1.5 (roughly 860M parameters) or even a distilled SDXL variant would likely produce comparable results for this style, with dramatically faster inference and lower GPU costs.

That's a future experiment. The optimization work here still transfers — snapshots, precision tuning, and memory layout apply regardless of model size. But the lesson is worth noting: match the model to the task, not the other way around.
